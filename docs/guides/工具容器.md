# 工具容器

## 为什么要用工具容器？

如果你想使用 `exec_container` 工具在容器内执行命令，**强烈建议使用专门的工具容器**，而不是直接在业务容器里执行命令。

- 🛡️ **安全隔离**：exec 操作只在工具容器内进行，不会影响跑着业务的容器
- 🧪 **放心折腾**：工具容器就是用来排障的，怎么折腾都不怕搞坏业务
- 🔧 **内置工具**：预装了各种排障工具（docker CLI、curl、dig、nc 等）
- 📁 **挂载排查**：通过挂载 Docker socket 和常用目录，可以从容器内排查宿主机 Docker 环境和文件问题

## 工具容器能做什么？

- 用 `docker` 命令查看宿主机上所有容器的状态、日志、配置
- 用 `curl` / `wget` 测试容器间网络连通性
- 用 `ping` / `traceroute` 排查网络问题
- 查看挂载的目录文件，排查权限、配置问题
- 安装临时工具做深度排障

## 使用步骤

1. 启动工具容器（配置文件在 `docker/docker-compose-toolbox.yml`）：
   ```bash
   docker compose -f docker-compose-toolbox.yml up -d
   ```

   > 💡 首次启动会自动安装常用排障工具，大约需要 30 秒~1 分钟。

2. 在 Docker-MCPilotS 的 Web UI 设置中开启「容器执行命令」功能

3. 给需要 exec 权限的 API Key 配置 scope 限制，**只允许对 `mcp-toolbox` 容器执行 exec**：
   ```yaml
   # secrets/auth.yaml 中对应 key 的 scope 配置
   scope:
     exec:
       include:
         - mcp-toolbox
   ```

4. 然后就可以让 Agent 通过 `exec_container` 工具在工具容器里排障了，比如：
   > "帮我用工具容器看看所有容器的运行状态"
   > "用 curl 测试一下 nginx 容器的 80 端口通不通"

## 关于构建方式

工具容器使用 **启动时自动安装工具** 的方式（方案 A），优点是：

- 通用：任何机器都能直接用，不需要预构建镜像
- 灵活：想加什么工具直接改 command 里的 apk add 列表
- 简单：一个 compose 文件搞定，没有额外的 Dockerfile

缺点是首次启动需要几十秒安装工具，之后重启就快了（只要容器不删，工具就在）。

## 挂载说明

工具容器默认挂载了 Docker socket（只读），所以容器内可以直接用 `docker` 命令操作宿主机的 Docker 环境。

如果不需要 docker 命令操作，可以去掉这个挂载。

可选挂载 NAS 常用目录（只读），方便排查文件问题：
```yaml
volumes:
  - /var/run/docker.sock:/var/run/docker.sock:ro
  - /volume1/docker:/data/docker:ro
  - /volume1/homes:/data/homes:ro
```

## 资源限制

默认限制 256MB 内存、0.5 核 CPU，排障够用了。需要的话可以自行调整。
